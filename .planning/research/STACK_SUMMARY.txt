KEY DECISIONS & RATIONALE
================================================================================

1. JWT over Sanctum for Mobile
   ❌ Sanctum tokens are opaque UUIDs (not JWTs)
   ✅ JWT enables standard refresh token cycle
   ✅ Mobile apps need access + refresh tokens (separated)
   ✅ tymon/jwt-auth is battle-tested

2. REST + GraphQL (Both!)
   ✅ REST for simple CRUD (mobile apps prefer simplicity)
   ✅ GraphQL for complex queries (avoid N+1 problems)
   ✅ OpenAPI docs for REST, Lighthouse schema for GraphQL

3. FHIR Support (Not Full HIPAA/GDPR Compliance in v3.0)
   ✅ Store medical data in structured schema
   ✅ Export as FHIR on-demand (for doctor sharing)
   ✅ Add FHIR codes/payloads to medical_records model
   ❌ Skip encryption/audit logging for now (v3.0 scope)

4. Reuse Existing Services/Policies
   ✅ API controllers call existing TripService, HealthService
   ✅ Policies work for both Filament + API
   ✅ No code duplication = less bugs

5. Rate Limiting Strategy
   ✅ Keep ApiRateLimitMiddleware (per-user by ID, or per-IP)
   ✅ Tune per-route: writes stricter than reads
   ✅ Use built-in Laravel throttle (no new package)

ANTI-PATTERNS TO AVOID
================================================================================

❌ Don't use Sanctum tokens for JWT flow
   → Use tymon/jwt-auth instead

❌ Don't expose raw Eloquent models
   → Use ApiResource/Fractal transformers

❌ Don't store medical records unencrypted
   → Use encrypt() on file storage (HIPAA requirement)

❌ Don't build custom JWT
   → Use battle-tested tymon/jwt-auth

❌ Don't commit secrets
   → Use .env (already in .gitignore)

❌ Don't skip rate limiting on writes
   → Throttle POST/PATCH endpoints strictly

❌ Don't return paginated data without meta+links
   → Mobile clients need explicit structure

❌ Don't skip API auth tests
   → APIs are invisible to manual testing

❌ Don't ignore CORS
   → Mobile apps need cross-origin access

❌ Don't duplicate health models
   → Extend existing (MedicalRecord, HealthRecord, Medication)

COMPARISON: REST vs GraphQL (When to Use Each)
================================================================================

REST (/api/v1/trips):
  ✅ Simple CRUD (create, read, list, update, delete)
  ✅ Mobile apps prefer simplicity
  ✅ Standard HTTP verbs (GET, POST, PATCH, DELETE)
  ✅ Easier caching (CDN-friendly, ETag support)
  ⚠️ Requires pagination + filtering design
  ⚠️ Can have N+1 problems with deep includes

GraphQL (/graphql):
  ✅ Complex queries (fetch exactly what you need)
  ✅ Solves N+1 problems
  ✅ Single endpoint (simpler mobile SDK)
  ✅ Self-documenting (schema is contract)
  ⚠️ Steeper learning curve
  ⚠️ Requires extra caching strategy (no HTTP caching)

Your Strategy: Offer Both
  - REST for mobile: simpler endpoints
  - GraphQL for complex: trips with nested itineraries + participants
  - Document both in /api/docs + Lighthouse schema
